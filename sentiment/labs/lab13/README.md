# Lab 13: Multi-Agent Architecture for Review Analysis

Ця лабораторна робота демонструє використання мультиагентної системи (Multi-Agent Crew) на базі локальної великої мовної моделі (LLM) для надійної обробки та екстракції даних.

## 1. Який use case
**Екстракція атрибутів з клієнтських відгуків.** Система отримує неструктурований текст (відгук, спам, короткі коментарі) і витягує з нього структурований JSON з такими полями: тональність (sentiment), аспекти, переваги (text), недоліки (text) та числова оцінка.

## 2. Які агенти є в Crew
Архітектура складається з 4 спеціалізованих компонентів:
1. **Triager Agent** — агент-маршрутизатор.
2. **Extractor Agent** — агент-витягувач даних.
3. **Reviewer Agent** — жорсткий валідатор (працює на правилах, без LLM).
4. **Fallback Handler (Repair Agent)** — обробник помилок.

## 3. Який workflow (Пайплайн)
1. Вхідний текст надходить до **Triager**, який визначає змістовність і задає маршрут (`full_schema`, `simple_schema` або `empty`).
2. **Extractor** формує JSON на основі тексту та маршруту.
3. Результат обов'язково перевіряється **Reviewer**.
4. Якщо є помилки, запускається **Fallback** (намагається виправити або зупиняє процес).
5. На виході формується фінальний лог зі статусом обробки.

## 4. Які delegation rules (Правила делегування)
* Усі запити спочатку йдуть до Triager.
* Якщо Triager визначає відгук як порожній/спам (`empty`), Extractor повертає порожній шаблон без звернення до LLM (економія ресурсів).
* Увесь згенерований Extractor-ом контент делегується на перевірку до Reviewer.
* Reviewer має право відхилити вихідні дані та примусово делегувати задачу до Fallback-агента для виправлення або ручного перегляду.

## 5. Як працює Reviewer
Reviewer виконує 3 рівні перевірки:
* **JSON Parsing:** Чи валідний синтаксис (відсутність зайвого тексту навколо JSON).
* **Schema Consistency:** Перевірка типів даних (наприклад, заборона масивів для полів `advantages` та `disadvantages`, вони мають бути рядками).
* **Logical Consistency:** Пошук суперечностей у даних (наприклад, `sentiment_type` — negative, але виставлена найвища оцінка).

## 6. Як працює fallback
Fallback має два механізми захисту:
* **Repair (Ремонт):** Якщо Reviewer знайшов структурну помилку (наприклад, масив замість рядка), Fallback-агент формує новий промпт із вказівкою на помилку і просить LLM перегенерувати JSON.
* **Safe Failure (Безпечний збій):** Якщо виявлено логічну суперечність (яку неможливо виправити програмно) або Repair провалився, Fallback зупиняє обробку і ставить статус `manual_review_required`. "Брудні" дані в базу не потрапляють.
