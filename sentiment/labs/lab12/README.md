# Lab 12: Tool-grounded Single-agent

**1. Який use case** Support assistant (помічник служби підтримки).

**2. Який agent task** Отримати текст звернення користувача, класифікувати тип проблеми, витягти необхідні сутності (email, номер замовлення), перевірити достатність даних через валідатор та сформувати фінальну відповідь на основі результатів цих інструментів.

**3. Які 2+ tools** Реалізовано 3 інструменти:
* `classify_ticket` — rule-based класифікатор категорії звернення;
* `extract_entities` — regex-екстрактор сутностей;
* `validate_required_fields` — перевірка наявності номера замовлення.

**4. Як запускати notebook** 1. Склонувати репозиторій локально або в середовище (наприклад, Google Colab).
2. Встановити залежності (`pip install transformers accelerate bitsandbytes`).
3. Відкрити файл `lab12_tool_grounded_single_agent.ipynb` та виконати всі комірки ("Run all").

**5. Де лежать logs** Всі логування викликів зберігаються у файлі `docs/tool_logs_lab12.jsonl`.

**6. Які test cases** Підготовлено 10 тестових сценаріїв, що охоплюють: прості запити, кейси з відсутніми даними (missing data), шумний текст (noisy text), неоднозначність (ambiguity/інша мова), а також нерелевантні тексти (подяка, технічний баг).

**7. Які метрики** * **Tool call success rate:** 100.0%
* **Average tool calls per task:** 3.0
* **Tasks with useful tool use:** 3
* **Unnecessary tool call count:** 5
* **Final answer correctness:** correct (4), partly correct (2), wrong (4)

**8. Головний висновок** Tool-grounded архітектура успішно вирішує проблему галюцинацій LLM (модель перестала вигадувати фактологічні дані, наприклад, ціни). Проте поточний жорсткий pipeline призвів до проблеми "Over-instruction": агент сліпо довіряв валідатору і вимагав номер замовлення у ситуаціях, де це не мало логічного сенсу (подяка, зміна пароля). Для покращення потрібна динамічна маршрутизація (Dynamic Routing) замість послідовного виклику всіх tools.
