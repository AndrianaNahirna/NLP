# Лабораторна робота №14: Оркестрація Stateful Flow для NLP Pipeline

## 1. Який use case обрано
**Класифікація запитів до служби підтримки (Support Ticket Classification).** Система автоматично приймає звернення користувачів, визначає їхній намір, класифікує за категоріями (Billing, Technical, General Inquiry) та гарантує стабільний структурований вивід.

## 2. Які етапи flow реалізовано
Пайплайн складається з 5 обов'язкових етапів та етапу обробки помилок:
1. `Ingest` — прийом сирого тексту, створення `case_id`, базове очищення та ініціалізація стану.
2. `Route` — визначення цільового маршруту на основі ключових слів та паттернів.
3. `Execute` — виконання класифікації (імітація роботи ML-моделі / агента) з генерацією міток та оцінки впевненості.
4. `Validate` — перевірка структури відповіді за схемою та фільтрація за порогом впевненості (confidence threshold).
5. `Fallback` — перехоплення відхилень, призначення ручного перегляду або безпечна відмова.
6. `Export` — формування фінального стабільного структурованого JSON-виводу.

## 3. Як виглядає state
Контекст кожного запуску керується об'єктом класу `FlowState` (dataclass). Його структура:
* `case_id`, `raw_text`, `clean_text`, `status`, `route`, `schema_name`, `routing_reason`;
* Словники проміжних даних: `tool_outputs`, `validation_result`, `fallback_result`, `final_output`;
* Списки для аудиту: `errors`, `warnings`, `fallback_triggered`, а також історія переходів `steps`.

## 4. Які routes підтримуються
* `support_classification` — для релевантних запитів до служби підтримки (фінанси, технічні збої, паролі).
* `unknown` — для нецільового або зашумленого вводу (out-of-domain запити).

## 5. Як працює validation
Етап валідації перевіряє відповідність результату виводу заданій JSON-схемі та наявність обов'язкових полів. Ключовий критерій успішності — рівень впевненості класифікатора (`confidence`). Якщо `confidence < 0.70`, валідатор маркує результат як невалідний і передає кейс на fallback.

## 6. Як працює fallback
Активується автоматично у разі помилки виконання або збою валідації. 
* При низькому confidence fallback робить частковий експорт (`partial_export`) і додає прапорець `needs_manual_review: true`.
* При невідомому маршруті (`unknown`) або фатальній помилці коду fallback виконує безпечну відмову проекту (`safe failure`), повертаючи статус `failed` без аварійної зупинки скрипта.

## 7. Який export format
Гарантований структурований формат **JSON**, який повертає чіткі поля (`case_id`, `status`, `prediction` або `reason`, `warnings`) незалежно від того, чи успішно пройшов запит, чи завершився збоєм.

## 8. Як запускати notebook
1. Відкрити файл `/notebooks/lab14_flow_orchestration_crewai_flows.ipynb` у Google Colab.
2. Запустити першу комірку для клонування репозиторію та встановлення залежностей із `requirements.txt`.
3. Виконати всі наступні комірки покроково (`Runtime -> Run all`), щоб прогнати тестові кейси, розрахувати метрики та згенерувати документацію.

## 10. Які метрики отримано
На базі 10 різноманітних тестових сценаріїв зафіксовано такі результати:
* **Flow completion rate:** 50%
* **Validation pass rate:** 40%
* **Fallback activation rate:** 50%
* **Fallback success rate:** 20%
* **Manual review / safe failure rate:** 60%
* **Export valid rate:** 50%
* **Average steps per case:** 5.2
* **Route accuracy:** 60%

## 11. Головний висновок
Перехід від ad-hoc лінійних скриптів до stateful flow кардинально підвищує стійкість системи до непередбачуваного та брудного вводу (edge cases). Завдяки чіткому розділенню зон відповідальності, ізольованому об'єкту стану та вбудованій системі жорсткої валідації з fallback-сценаріями, NLP-пайплайн стає повністю контрольованим, передбачуваним та готовим до інтеграції у реальні сервіси.
