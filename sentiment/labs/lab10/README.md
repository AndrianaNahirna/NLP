# Лабораторна робота 10: NER Pipeline & Hybrid Rules

**1. Який корпус / evaluation set використано**
Використано очищений корпус відгуків клієнтів про українські інтернет-магазини електроніки (Rozetka, Citrus). Для оцінки (evaluation set) було вручну підготовлено 20 репрезентативних речень, що містять як стандартні сутності (імена, локації), так і специфічні доменні дані (номери замовлень, ціни).

**2. Який pipeline запущено**
Запущено стандартний NER pipeline бібліотеки **spaCy** з використанням базової української моделі `uk_core_news_sm`.

**3. Які rules додано**
До базової моделі додано гібридний шар `EntityRuler` із наступними правилами:
* **ORDER_ID**: розпізнавання за регулярними виразами (символ «№» або слово «замовлення» + цифри).
* **MONEY**: поєднання числових токенів із назвами валют («грн», «баксів», «гривень»).
* **ORG**: словниковий метчинг для ключових брендів (Розетка, Цитрус, Монобанк, ПриватБанк) та служб доставки.
* **DATE**: паттерн для числових форматів дат (наприклад, 10.10.2023).

**4. Що baseline знаходив добре**
Базова модель без додаткових налаштувань успішно справлялася з ідентифікацією імен людей (`PER`) та назв міст чи вулиць (`LOC`).

**5. Що baseline пропускав**
Baseline повністю ігнорував номери замовлень та грошові суми. Також спостерігалися помилки типу (Type Errors): наприклад, назва «Розетці» часто тегувалася як людина (`PER`), а «Нову Пошту» модель вважала локацією (`LOC`).

**6. Що rules покращили**
Гібридний підхід дозволив досягти 100% recall для сутностей `MONEY` та `ORDER_ID` у тестовому наборі. Правила також виправили помилкову типізацію доменних брендів та забезпечили стабільне розпізнавання дат у числовому форматі.

**7. Які проблеми ще не вирішені**
Найскладнішою проблемою залишається екстракція сутностей класу `PRODUCT` (конкретні моделі техніки, як-от «iPhone 13 Pro»), оскільки вони мають високу варіативність назв. Також модель все ще пропускає складні часові інтервали, описані природною мовою.
