# LPNU NLP Lab #4: Rule-Based Information Extraction

## 1. Типи полів, які витягуються
В рамках цієї лабораторної роботи реалізовано витяг трьох типів сутностей із текстів відгуків:
* **`DATE` (Дати):** Точні календарні дати покупки або доставки. Нормалізуються до формату `YYYY-MM-DD`.
* **`AMOUNT` (Грошові суми):** Ціни на товари або суми транзакцій. Нормалізуються до формату `value (float) + currency (ISO)`, наприклад, `499.0 UAH`.
* **`LOCATION` (Міста):** Назви українських міст (місця доставки/проживання). Нормалізуються до базової форми (Називний відмінок з великої літери, наприклад, `Львів`).

## 2. Використані правила та словники
Для забезпечення підходу *Precision-first* було використано комбінацію регулярних виразів, словників та логічних перевірок:
* **Регулярні вирази (Regex):** Використовуються жорсткі межі слів (`\b`), щоб уникнути витягу частин інших слів, а також підтримка плаваючої коми для сум (`[.,]\d{1,2}`).
* **Логічні обмеження (Python):** Додано перевірки валідності дат (місяці `1-12`, дні `1-31`), щоб відсіяти неіснуючі дати або версії програм (напр., `00.05.2019`).
* **Словник `months_ua.json`:** Для мапінгу текстових українських назв місяців (у різних відмінках) у числові еквіваленти.
* **Словник `currencies.json`:** Для зведення різних написань валют (`грн`, `гр`, `₴`, `$`, `usd`) до єдиного стандарту (`UAH`, `USD`).
* **Словник `locations.json`:** Для нормалізації відмінків найпопулярніших міст України (`у Києві`, `до Львова`) до базової форми (`Київ`, `Львів`).

## 3. Precision по кожному типу поля
Оцінка проводилася на еталонній вибірці (Gold Subset), що складається з 30 репрезентативних текстів:

| Тип поля (Field Type) | Precision |
| :--- | :---: |
| **DATE** | 1.00 (100%) |
| **AMOUNT** | 1.00 (100%) |
| **LOCATION** | 1.00 (100%) |

*Примітка: Завдяки жорстким правилам та словникам ми уникнули хибних спрацювань (False Positives) на еталонному датасеті.*

## 4. Топ-5 edge cases у корпусі
Під час розробки на всьому датасеті було виявлено та опрацьовано такі граничні випадки:
1. **Версії прошивок/ПЗ:** Рядки типу `2.0.12` алгоритм міг сприйняти як дату `2012-00-02` (вирішено логічними перевірками діапазонів).
2. **Дробові суми з комою:** В українській мові часто пишуть `1,5 грн` замість `1.5` (вирішено розширенням regex-правила та заміною коми на крапку перед приведенням до float).
3. **Відсотки та кількість:** Рядки `100%` або `2 шт` сприймалися як суми (вирішено жорстким словником валют — число без ідентифікатора валюти поруч ігнорується).
4. **Прикметники від міст:** Словосполучення типу `київський торт` помилково витягували місто (вирішено строгими межами слова `\b` у regex та відсутністю прикметників у словнику).
5. **Неповні дати (номери моделей):** Текст `модель 10.12` витягувався як `2012-10` (вирішено вимогою обов'язкової наявності року після другої крапки).

## 5. Плани на покращення у наступних лабах
* **Інтеграція ML/NER-моделей:** Використати моделі на базі spaCy або трансформерів для витягу складніших сутностей (ПІБ, назви брендів), де правила і словники безсилі.
* **Feature Engineering:** Дослідити, як наявність певних сутностей впливає на тональність (наприклад, перевірити гіпотезу, чи частіше негативні відгуки містять згадки про конкретні міста або ціни).
* **Боротьба з дисбалансом:** Застосувати техніки балансування класів для покращення якості основної задачі класифікації тональності.
