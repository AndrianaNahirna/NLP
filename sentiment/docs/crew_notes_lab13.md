# Crew Notes: Lab 13 (Multi-Agent Architecture)

### 1. Який use case обрано
Екстракція структурованих атрибутів з клієнтських відгуків. Система витягує тональність (sentiment), згадані аспекти товару/послуги, текстові описи переваг та недоліків, а також числову оцінку.

### 2. Які агенти є в crew
У нашому пайплайні працюють 4 компоненти:
1. **Triager Agent** (Маршрутизатор)
2. **Extractor Agent** (Екстрактор)
3. **Reviewer Agent** (Валідатор)
4. **Fallback Handler / Repair Agent** (Обробник збоїв)

### 3. Роль кожного агента
* **Triager**: Аналізує вхідний текст, визначає його змістовність і задає маршрут (`full_schema`, `simple_schema` або `empty` для спаму/порожніх повідомлень).
* **Extractor**: Безпосередньо витягує дані з тексту у форматі JSON відповідно до вказівок Triager-а.
* **Reviewer**: Виконує жорсткий контроль якості результатів Extractor-а. Не використовує LLM, спирається на чіткі правила перевірки (rule-based).
* **Fallback**: Використовує LLM для виправлення помилок схеми (Repair) або генерує безпечний збій (Safe Failure) з вимогою ручної перевірки.

### 4. Правила делегування
1. Вхідний текст завжди спочатку потрапляє до **Triager**.
2. Результат (маршрут) передається **Extractor**. (Якщо маршрут `empty`, Extractor відразу повертає порожній шаблон без звернення до LLM).
3. Згенерований JSON обов'язково передається **Reviewer**.
4. Якщо Reviewer повертає `accept`, робота завершується.
5. Якщо Reviewer повертає `repair_needed`, задача делегується **Fallback (Repair)**.
6. Якщо Reviewer повертає `fallback_needed`, управління переходить до **Fallback (Safe Failure)**.

### 5. Що перевіряє Reviewer
* **Валідність JSON:** чи немає помилок парсингу.
* **Типізацію (Schema Consistency):** чи не повернула модель масиви (arrays) для полів `advantages` та `disadvantages` замість об'єднаних рядків (strings).
* **Логічну цілісність (Logical Consistency):** чи немає суперечностей (наприклад, `sentiment_type: "negative"`, але `rating_mentioned: 5`).

### 6. Коли спрацьовує fallback
* **Repair Agent** спрацьовує, коли порушена структура даних (наприклад, LLM згенерувала масив або вставила текст поза JSON).
* **Safe Failure (Manual Review)** спрацьовує при логічних суперечностях або якщо Repair Agent не зміг полагодити схему з першої спроби.

### 7. Що crew покращив порівняно з single-agent
Single-agent (Baseline) сліпо вірив своїм галюцинаціям: він зберігав масиви замість рядків (при ін'єкціях у промпт) і пропускав логічні суперечності. Crew-підхід заблокував ці аномалії. Завдяки Reviewer та Fallback, фінальна база даних залишається чистою від невалідних форматів та суперечливих даних.

### 8. Де multi-agent підхід був зайвий
Для 80% стандартних, чітких і грамотно написаних відгуків (наприклад: *"Кросівки супер, доставка швидка, 5/5"*) використання мультиагентної системи було надлишковим. Якісний базовий промпт в Extractor-і чудово справлявся сам, а інші агенти лише додавали мілісекунди до часу обробки (хоча й не робили зайвих запитів до LLM).

### 9. Які помилки залишилися
* **Втрата нюансів при складному сарказмі:** Іноді модель може неправильно оцінити дуже тонкий сарказм, якщо в тексті немає явних тригерів, які б Reviewer міг відловити як суперечність.
* **Обмеженість Repair-циклу:** Наразі є лише одна спроба ремонту. Якщо Repair Agent знову помилиться, система відразу падає в Safe Failure.

### 10. Що б ви фіксили далі
1. Додала б кілька ітерацій для Repair Agent (наприклад, `max_retries = 3`).
2. Впровадила б механізм `Self-Reflection` для Extractor-а, щоб він сам перевіряв свій JSON перед відправкою до Reviewer-а.
3. Розширила б логіку Reviewer-а для виявлення більшої кількості неочевидних суперечностей (наприклад, порівняння знайдених `advantages` із загальним `sentiment`).
