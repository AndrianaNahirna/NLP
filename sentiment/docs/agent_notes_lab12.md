# Agent Notes (Lab 12)

**1. Який use case обрано**
Обрано "Варіант 1: Support assistant".

**2. Який agent task**
Агент отримує текст звернення користувача, класифікує тип проблеми (issue type), витягує сутності (наприклад, email або номер замовлення), перевіряє наявність потрібних полів та формує коротку відповідь служби підтримки.

**3. Які tools реалізовано**
Реалізовано 3 інструменти у вигляді callable-функцій у коді:
* `classify_ticket(text)` — rule-based класифікатор категорій звернень.
* `extract_entities(text)` — regex-екстрактор для витягу email та order_id.
* `validate_required_fields(extracted_data)` — валідатор для перевірки наявності номера замовлення.

**4. Як агент вирішує, коли викликати tool**
Для максимального контролю в цій системі реалізовано послідовний (детермінований) алгоритм замість вільної автономності. Агент послідовно викликає інструменти один за одним (`classify` -> `extract` -> `validate`), а їх результати формують grounded prompt для генерації фінальної відповіді.

**5. Як логуються tool calls**
Усі виклики логуються через окремий модуль та зберігаються у файл `tool_logs_lab12.jsonl` у форматі JSON Lines. Для кожного виклику записуються: `timestamp`, `task_id`, `tool_name`, `input`, `output`, `success` та `error`.

**6. Що tools реально покращили**
Інструменти суттєво зменшили фактологічні галюцинації. Наприклад, у задачі із запитом ціни доставки baseline-модель вигадала ціну 150 грн, тоді як tool-grounded агент спирався на дані систем і не став вигадувати неправдиву інформацію.

**7. Де tools були зайві або не допомогли**
У половині тестових кейсів інструмент валідації (і наступне прохання надати номер замовлення) був зайвим. Це ставалося в нерелевантних сценаріях: коли користувач просив змінити пароль, скаржився на виліт додатка або просто залишав подяку.

**8. Які помилки лишилися**
* **Over-trusting the tool:** LLM занадто жорстко слідує валідатору і вимагає номер замовлення там, де це порушує логіку діалогу.
* **Недосконалість rule-based підходу:** Класифікатор `classify_ticket` не спрацював на слові "оплатив", оскільки шукав точний збіг слова "оплата", що призвело до неправильної маршрутизації звернення.

**9. Що б ви фіксили далі**
Варто додати динамічну маршрутизацію: агент має викликати інструмент `validate_required_fields` ТІЛЬКИ якщо класифікатор попередньо визначив категорію `billing`. Також слід замінити жорсткі `if/else` правила у класифікаторі на використання лематизації, ембедингів або невеликої LLM для розуміння семантики.
