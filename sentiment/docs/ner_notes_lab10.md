# NER Notes: Lab 10

**1. Який pipeline вибрано**
Використано базовий pipeline від spaCy з українською моделлю `uk_core_news_sm`.

**2. Які сутності він уміє з коробки**
Стандартна модель розпізнає базові класи: `PER` (імена), `LOC` (географічні локації) та `ORG` (організації).

**3. Які доменні сутності для вашого корпусу важливі**
Для e-commerce корпусу (відгуки про магазини електроніки) критичними є:
* `ORDER_ID` (номери замовлень)
* `MONEY` (грошові суми, ціни)
* `ORG` (назви конкретних локальних брендів і банків)
* `PRODUCT` (назви та моделі техніки)
* `DATE` (дати покупок чи звернень у сервіс)

**4. Які правила ви додали**
Через компонент `EntityRuler` перед стандартним NER було додано 4 типи правил:
* Regex-патерни для `ORDER_ID` (напр., символ "№" + цифри).
* Regex-патерни для `MONEY` (цифри + "грн", "баксів").
* Словникові патерни для `ORG` (конкретні назви: Розетка, Цитрус, Нова Пошта).
* Regex-патерн для числових `DATE` (dd.dd.dddd).

**5. Що стало краще після правил**
Гібридний підхід повністю закрив проблему витягування цін (`MONEY`) та номерів тікетів/замовлень (`ORDER_ID`), забезпечивши 100% знаходження на тестовому наборі. Також були усунуті хибні спрацьовування (type errors), коли бренди плуталися з іменами людей або локаціями.

**6. Які помилки залишилися**
* Пропуски сутностей `PRODUCT`: модель не розуміє складні англомовні назви техніки (наприклад, "iPhone 13 Pro").
* Помилки меж (`boundary errors`) у складних текстових датах: модель може знайти день і місяць, але відсікти рік.

**7. Що б ви фіксили далі**
Наступним логічним кроком є додавання великих словників (Gazetteers) через `PhraseMatcher` для охоплення бази товарів (смартфони, ноутбуки, аксесуари), щоб вирішити проблему з класом `PRODUCT`.
