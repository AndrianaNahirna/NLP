# Memory & Knowledge Policy - Lab 14

## 1. Що зберігається в state
У стані (`FlowState`) зберігається виключно короткочасний контекст поточного прогону (case-level data):
* Унікальний ідентифікатор кейсу (`case_id`).
* Вхідні дані (`raw_text`, `clean_text`).
* Дані маршрутизації (`route`, `schema_name`, `routing_reason`).
* Масиви зібраних результатів (`tool_outputs`, `validation_result`, `fallback_result`, `final_output`).
* Масиви логів та історії кроків (`errors`, `warnings`, `steps`).

## 2. Що не зберігається
* Довготривала інформація про користувача між різними сесіями.
* Глобальні стани, які можуть вплинути на обробку інших запитів.
* Оригінали великих документів, якщо для подальшої обробки потрібна лише їх витяжка (щоб не перевантажувати пам'ять).

## 3. Які intermediate outputs можуть передаватися між етапами
Між етапами передаються лише структуровані проміжні результати через спеціальні словники в об'єкті стану:
* Дані з етапу `execute` передаються у `validate` через поле `tool_outputs`.
* Висновки етапу `validate` передаються у `fallback` або `export` через поле `validation_result`.
* Ці проміжні виводи вважаються "чернетками", поки не пройдуть валідацію.

## 4. Як логуються помилки
Помилки (як технічні збої екзекуції, так і логічні помилки валідації) не ламають програму. Вони:
* Додаються у масиви `errors` або `warnings` всередині `FlowState`.
* Записуються в історію кроків (`steps`) з відповідним статусом (наприклад, `status: error`).
* Невалідні виводи екзекутора записуються як помилки, а не приймаються як істина.

## 5. Як працює knowledge / schema registry
Система використовує схеми валідації (наприклад, `ticket_classification_schema`) як зовнішній довідник. Етап `Validate` звіряє `tool_outputs` із цією схемою, щоб переконатися у наявності всіх обов'язкових полів (наприклад, `predicted_class` та `confidence`) та правильності їхніх типів.

## 6. Які файли є read-only knowledge
* Словники ключових слів для роутингу (наприклад, списки слів "оплата", "пароль").
* Схеми валідації (JSON Schemas).
* Ці ресурси є виключно "read-only" — пайплайн читає їх для прийняття рішень, але ніколи не модифікує під час роботи.

## 7. Як запобігається state pollution (забруднення стану)
Для кожного нового текстового запиту етап `Ingest` створює абсолютно новий екземпляр класу `FlowState` з новим `case_id`. Дані з попередніх запусків ніяк не можуть потрапити в поточний стан, що гарантує повну ізоляцію кейсів.

## 8. Які дані не можна логувати
* API-ключі (OpenAI, Gemini тощо).
* Паролі, токени авторизації та облікові дані баз даних.
* Надлишкові конфіденційні персональні дані користувачів (PII), якщо вони не є критично необхідними для фінального експорту.

---

### Summary
* State stores only case-level data needed for processing.
* Raw input is stored for the current run only.
* Invalid intermediate outputs are logged as errors, not used as truth.
* Knowledge files are read-only.
* Schemas are used for validation and cannot be modified by the flow.
* API keys and credentials are never stored in state or logs.
