# Flow Notes - Lab 14

## 1. Який use case обрано
Класифікація запитів до служби підтримки (Support Ticket Classification). Flow отримує текст звернення і визначає його категорію (Billing, Technical тощо) або відхиляє нерелевантні повідомлення.

## 2. Які етапи flow
Пайплайн складається з 5 основних етапів та 1 резервного:
1. `Ingest` (прийом та очищення)
2. `Route` (вибір маршруту)
3. `Execute` (виконання класифікації)
4. `Validate` (перевірка якості та структури)
5. `Fallback` (спрацьовує за потреби для обробки помилок)
6. `Export` (фінальне форматування JSON)

## 3. Яка структура state
Використовується об'єкт `FlowState` (dataclass), який зберігає короткочасний контекст поточного запуску. 
Ключові поля: `case_id`, `raw_text`, `clean_text`, `status`, `route`, `tool_outputs`, `validation_result`, `fallback_result`, `final_output`, масиви `errors` та `warnings`, а також масив `steps` для логування історії переходів.

## 4. Які routes можливі
* `support_classification` — для запитів, що містять ключові слова, пов'язані з проблемами користувачів (оплата, пароль, помилки).
* `unknown` — для запитів, які не стосуються підтримки (наприклад, питання про погоду).

## 5. Що робить execute
На цьому етапі імітується робота класифікатора (mock). Функція перевіряє текст на специфічні паттерни та генерує об'єкт із трьома полями: `predicted_class`, `confidence` (рівень впевненості) та `explanation`. У реальному проєкті тут здійснювався б виклик ML-моделі або агента CrewAI.

## 6. Що перевіряє validate
Валідатор перевіряє наявність усіх обов'язкових полів у результаті (`predicted_class`, `confidence`). Головна перевірка — чи перевищує показник `confidence` поріг у 0.70. Якщо впевненість нижча, валідація не проходить.

## 7. Коли спрацьовує fallback
Fallback активується у трьох випадках:
* Якщо роутер обрав `unknown` маршрут (відбувається безпечна відмова).
* Якщо під час `execute` сталася системна помилка коду.
* Якщо етап `validate` повернув помилку (наприклад, через низький `confidence`). У цьому випадку fallback маркує запит як `Needs Manual Review`.

## 8. Який export format
Стабільний структурований JSON. Завжди містить `case_id`, `status` та `warnings`. Для успішних кейсів додається блок `prediction`. Для неуспішних — причину відмови (`reason`) та прапорець `needs_manual_review`.

## 9. Що flow покращив порівняно з ad-hoc pipeline
* **Ізоляція збоїв:** Помилка на етапі виконання чи валідації не "кладе" всю програму.
* **Дебагинг:** Завдяки масиву `steps` видно весь життєвий цикл запиту.
* **Стабільність інтерфейсу:** Зовнішні системи завжди отримують передбачуваний JSON, а не StackTrace помилки Python.

## 10. Де flow був надлишковим
Для абсолютно тривіальних та чітких запитів (наприклад, повідомлення з одного слова "оплата") створення об'єкта стану, роутинг та багатокрокова валідація додають оверінжинірингу. Звичайний if/else впорався б швидше і з меншими витратами пам'яті.

## 11. Що б ви фіксили далі
1. Додати етап `Repair` перед `Fallback`: якщо валідація не пройшла через відсутність поля, спробувати попросити модель перегенерувати відповідь.
2. Додати таймстемпи (timestamps) у масив `steps`, щоб моніторити тривалість кожного етапу.
